## Epic 1. 듀얼 추천 엔진 (사먹기 vs 해먹기)

### 1. 사먹기(외식) 모드: 2단계 추천 구조 (2026-04-23 개편)

#### 1.1 추천 구조 설계 원칙

추천은 **메뉴 추천**과 **식당 추천** 두 단계로 분리한다. 각 단계는 서로 다른 데이터 소스를 기반으로 동작한다.

```
1단계: 메뉴 추천 (무엇을 먹을지)
  └── 기준: 유저의 메뉴 선택 이력 데이터
  └── 결과: 메뉴명 카드 5~7개

          ↓ 카드 클릭 시

2단계: 식당 추천 (어디서 먹을지)
  └── 기준: 유저의 입맛 평가 데이터 기반 매칭률 순
  └── 결과: 해당 메뉴 취급 식당 목록 (아코디언 확장)
```

#### 1.2 메뉴 추천 (1단계)

- **추천 기준:** 유저의 메뉴 선택 이력 데이터 기반
- **노출 수량:** 5~7개 고정
- **카드 구성:** 메뉴명만 표시. 썸네일 이미지 없음.
- **순서 의미:** 상단 카드가 1순위 추천을 의미하지 않음. 유저 데이터 기반 적합도 순 배열.

#### 1.3 식당 추천 (2단계 - 아코디언 확장)

- **트리거:** 메뉴 카드 클릭
- **추천 기준:** 유저의 입맛 평가 데이터 기반 매칭률 순
- **표시 정보:** 메뉴명 / 식당명 / 가격 / 도보 이동 시간(분)
- **닫기:** 카드 재클릭 시 닫힘

#### 1.4 외부 연결 (식당 행 클릭 시)

- **연결 대상:** 네이버 지도 해당 식당 페이지 (딥링크)
- **목표 랜딩:** 식당 내 메뉴 탭 (네이버 플레이스 ID 활용)
- **제공 정보:** 네이버 지도 내 메뉴 이미지, 가격, 리뷰 등 위임
- **추후 확장:** 카카오/구글 지도 선택 옵션 추가 예정

> ⚠️ **미확정:** 네이버 지도 딥링크에서 특정 메뉴 항목까지 직접 랜딩 가능 여부 — 추가 기술 검토 필요

#### 1.5 이 구조를 채택한 이유

정확한 메뉴 이미지, 리뷰 등의 데이터를 자체 크롤링/저장하는 것은 각 플랫폼의 이용약관상 제약이 있다. 네이버/카카오/구글 지도의 기존 인프라에 위임함으로써 이 제약을 우회하고, 서비스는 "무엇을 먹을지 결정"이라는 핵심 가치에 집중한다.

### 2. 위치 기반 반경 제어 정책

- **초기 설정:** 유저의 현재 위치 기준, 도보 이동이 용이한 거리(반경 1km 내외)를 디폴트로 노출
- **거리 제어 UI:** 직관적인 거리 필터 칩(Chip) 제공 (1km, 2km, 3km 등) → 유저가 점진적으로 반경 확장 탐색
- **도보 시간 표시:** 거리(km) 대신 도보 이동 분 단위로 표시 ✅ 확정

#### 2.1 초정밀 매칭 예외 규칙 (Perfect Match Exception)

- 유저의 입맛 데이터와 일치율이 극도로 높은 식당의 경우, 설정된 거리 필터를 약간 벗어나더라도 리스트 하단에 노출
- 카피: "거리는 좀 있지만, 입맛에 완벽한 곳이에요!" → 선택권은 유저에게 위임

> ⚠️ **미확정 (R-1):** 완벽 매칭 예외 규칙 발동 기준 수치 ("매칭률 95% 이상" — 확정 전)
> ⚠️ **미확정 (R-2):** 거리 필터 예외 시 허용 초과 범위 (얼마나 더 멀리까지 노출할지 — 확정 전)

### 3. B2B 광고 노출 및 지도 핀(Pin) UI 정책

광고(비용 지불) 식당이 무분별하게 노출되지 않으며, 유저의 입맛 데이터 부합도에 따라 핀의 형태와 노출 방식을 차등화하여 신뢰도 유지.

| 유형 | 조건 | UI |
|---|---|---|
| 오가닉 | 비용 미지불 + 유저 입맛 매칭 | 일반 원형(Circle) 마커 핀 |
| 타겟팅 광고 | 비용 지불 + 유저 입맛 매칭 | 별표(Star) 마커 핀 (자연스러운 우선 노출) |
| 챌린지 광고 | 높은 비용 지불 + 약간의 불일치 | 반짝이는 별표(Sparkling Star) 핀 + 위트 카피 팝업 |

- **챌린지 광고 카피:** "도전장을 받아들이시겠어요? (당신의 입맛을 만족시킬 준비가 된 식당)"
- **유저 제어권:** '도전하는 식당 제외하고 보기' 토글로 즉시 비활성화 가능

### 4. 해먹기(요리) 모드: 점진적 재고 파악 및 결품 대응

**4.1 결품 체크 UI**
- 레시피 상세 화면 내 [이 재료가 없어요] 버튼 배치 → 유저가 부족한 식재료 선택

**4.2 결품 발생 시 분기 처리 (Action Routing)**

| 옵션 | 내용 |
|---|---|
| Option A (B2B 연계 구매) | 근처 마트 픽업 또는 동네 마트 즉시 배달 연동 → 해당 재료 장바구니 연결 |
| Option B-1순위 | 해당 재료 없이도 조리 가능한지 판단 후 그대로 진행 안내 |
| Option B-2순위 | 완벽히 대체 가능한 다른 식재료 제안 (예: 소고기 대신 돼지고기) |
| Option B-3순위 | 결품된 중요 재료가 들어가지 않는 새로운 메뉴로 재추천 |

### 5. 재고 데이터 자동 추적 및 관리 (Automated Inventory Tracking)

**5.1 암묵적 재고 차감 (Implicit Deduction)**
- B2B 연동 구매 식재료 → 유저의 '가상 냉장고 DB'에 자동 적재
- 유저가 특정 레시피로 '실제 조리(식사 완료)' 액션 시 → 백엔드에서 해당 레시피 사용 정량만큼 자동 차감
- **알림 정책:** 특정 식재료 소진 임박 시 알림 (예: "계란이 4개 남았어요! 오늘 저녁에 채워둘까요?")

**5.2 수동 입력 지원 (Optional)**
- 마이페이지 내 '나의 식재료/냉장고' 메뉴 → 유저가 자발적으로 재고 점검 및 수정 가능. 강제 입력 없음.

### 6. 유저 페르소나 추정 및 B2B 업셀링

**6.1 레시피 선택 기반 라이프스타일 추정**

| 레시피 선택 패턴 | 추정 결과 |
|---|---|
| 손이 많이 가는 정통 한식 위주 | 기본 조미료 및 식재료가 풍부한 다인/가족 가구 |
| 간편식(간장계란밥 등) 위주 | 기본 재료만 보유한 1인/자취 가구 → 추천 풀(Pool) 조정 |

> ⚠️ **미확정 (R-5):** 가구 형태 자동 추정 알고리즘 구체 기준 및 구현 방식

## Epic 2. 다차원 입맛 평가 시스템

### 1. 평가 유도 노출 정책 (Non-blocking)

| 유형 | 조건 | UI/UX |
|---|---|---|
| 푸시 알림 | 메뉴 추천 액션 발생 후 **2시간 경과** 시점 ✅ 확정 | "오늘 추천해 드린 [메뉴명], 입맛에 잘 맞으셨나요?" |
| 인앱 팝업 | 식사 완료 예상 시간 이후 앱 재진입 시 | 화면 하단 Bottom Sheet 또는 상단 Toast 메시지 (전면 모달 절대 지양) |
| 자율 기록 | 상시 | '나의 식사 기록' 탭 — 앱 추천 없이도 자유롭게 메뉴 검색/입력 후 평가 가능 |

### 2. 평가 UX — 2단계 구조 (Progressive Disclosure)

**Step 1. 메인 평가 (Mandatory)**
- UI: 3버튼 형태
  - [내 입맛 아님] - [그냥저냥] - [완전 내 스타일]
- 이 단계까지만 완료해도 '평가 완료'로 간주하여 데이터 저장

**Step 2. 상세 평가 (Optional)**
- UI: 메인 평가 완료 직후 [상세 평가 남기기 🔽] 아코디언 메뉴 또는 추가 버튼 노출
- 동기 부여: "상세 평가를 남길수록 다음번 추천 성공률이 더 높아져요!" 마이크로 카피

### 3. 다차원 슬라이더 동작 정책

- **개수 제한:** 한 메뉴당 상세 평가 슬라이더 **최대 5개** ✅ 확정
- **스케일 구조 (Zero-centered):** 가운데(정중앙) = '나에게 완벽하게 맞는 상태'
  - 좌측 이동: 해당 맛/속성이 매우 적음
  - 우측 이동: 해당 맛/속성이 매우 강함

### 4. 평가 속성 구성 정책

- **동적 속성 (Dynamic):** 음식 카테고리에 따라 시스템이 가장 적합한 핵심 평가 속성 자동 매칭 및 노출
- **사용자 정의 속성 (Custom):** [➕속성 추가] 기능으로 유저가 직접 속성 추가 가능
- **Phase 2 보류:** 한식/정식류의 반찬 구성 평가 로직 → 2차 고도화 스펙으로 분리

### 5. 부정 평가 데이터 처리 정책 (Negative Feedback Logic)

**정책 의도:** '메뉴 자체'가 싫은 것인지, '해당 식당의 조리 방식'이 싫은 것인지를 분리하여 판단.

- 특정 식당에 부정 평가 → 영구 블랙리스트 처리 없음
- 입맛 유사도가 높은 다른 유저들의 평가 데이터를 교차 검증(협업 필터링)
- 결과: 비슷한 입맛 그룹이 호평한 식당 → 자연스럽게 상위 랭크업

> ⚠️ **미확정 (R-6):** 입맛 유사도 협업 필터링 구체적 구현 방식 및 유사도 산정 알고리즘

## Epic 8. 초개인화 동적 정렬(Dynamic Sorting) 및 마이크로 태그

### 1. 마이크로 취향 가중치 폭격 로직 (Micro-Preference Weighting)

**1.1 메뉴의 벡터화 (태그 쪼개기)**

- 예시: [#메인: 짜장면], [#토핑: 반숙 계란후라이], [#사이드: 군만두]

**1.2 가중치 펌핑 (Weighting Pump)**
- 특정 식재료에 강한 선호 데이터를 가진 유저 → 해당 태그 포함 메뉴/식당에 강력한 추가 가점(+) 부여 → 최상단 노출
- 평점이나 거리를 일부 무시하고 우선 노출

### 2. 라이프스타일 기반 시간표 밸런싱 (Time-based Meal Balancing)

**2.1 직전/이후 식사 연계 필터링**
- 저녁에 헤비한 식사 패턴 인지 → 당일 점심은 가벼운 메뉴에 가중치 부여

> ⚠️ **미확정 (R-4):** 직전/이후 식사 연계 필터링 구체적 구현 방식 및 데이터 소스

**2.2 혈당 및 생체 리듬 보호 (Fasting & Health Guard)**
- 긴 공복 이후 첫 끼니 → 혈당 스파이크 방지 메뉴에 가중치 부여

> ⚠️ **미확정 (R-3):** 혈당 스파이크 방지 가중치 구현 방식 및 데이터 소스

### 3. UI 시각화 및 UX 라이팅 (Persona Integration)

**동적 정렬 발동 시 팝업/배너 텍스트 예시:**
> 💬 "대표님, 오늘 저녁에 한식 드실 텐데 점심은 가벼운 면 요리 어떠십니까? 마침 이 근처 라멘집에 대표님이 사랑하시는 '완벽한 반숙란'이 올라간다고 하네요. 기왕이면 여기로 가시죠! (찡긋)"
